## 当前方向

RemoteLab 不是“远程和模型聊天”的工具，也不应该主要服务已经非常会用 AI 的少数人。

它更像：

- 一个把真实机器变成 AI 自动化执行面的产品
- 一个帮助用户把模糊问题澄清成可执行方案的协作系统
- 一个让用户从“反复亲手做琐事的人”变成“定义目标、校验结果的人”的产品

它的第一阶段目标很直接：

- 让用户花很短的时间
- 就能把一个原本每周都要花几个小时的重复数字工作交给 AI
- 并且在手机与桌面之间自然切换，由真实机器把事情做完

---

## 核心判断

### 1. 最值得解决的问题不是并发，而是“什么事情值得被自动化”

很多真实世界的高频痛点并不复杂：

- 数据整理
- 简单分析
- 报表生成
- 文件批处理
- 导出导入
- 通知与同步

这些问题往往不是需要“多开几十个 session”，而是需要一个系统帮用户快速识别：这事可不可以自动化、需要补什么输入、最小可行方案是什么。

### 2. 目标用户默认不是 AI-native，也不是会自己给 AI 派活的产品经理

很多潜在用户：

- 以前没系统用过研发提供的自动化
- 不知道 agent 工作模式会带来什么变化
- 也不一定能一上来就把需求拆成清晰 prompt

所以 RemoteLab 里的 AI 不能只等一个“写得很漂亮的需求”；它要主动承担一部分产品经理 / 方案设计者的责任，帮用户：

- 把问题讲清楚
- 把样例和约束补齐
- 判断该用脚本、数据处理、定时任务还是别的方式
- 在执行前先收敛成一个清晰的 automation brief

### 3. 首屏应该降低启动成本，而不是把用户丢进一个空的 session list

如果目标是让普通用户快速开始用好 RemoteLab，那么 fresh install / first open 的默认体验就非常关键。

对于这类用户来说，一个空白的 session list 往往意味着：

- 我现在该从哪里开始？
- 这个东西到底能帮我干什么？
- 我应该先提需求、上传文件，还是先创建点什么？

更合理的默认应该是一个内建的 `Welcome App`，它承担的不是“教程页”角色，而是一个轻量的 AI 引导员：

- 用非常短的话说明自己能帮助处理哪些常见重复数字工作
- 引导用户补充自己的角色背景、当前工作环境和最想先解决的问题
- 帮用户从模糊诉求里挑出一个最值得先做的 automation 切入口
- 继续把这些信息收敛成第一份可执行的 automation brief

这里的关键不是收集一堆 profile 字段，而是帮助用户尽快拿到第一次真实收益，并顺带建立后续更懂他的上下文。

### 4. 手机 + 桌面 + 真机执行是核心优势，不是附属特性

RemoteLab 的价值不是“聊天能跨端同步”，而是：

- 用户可以在手机上随手发问题、截图、样例文件和补充说明
- AI 可以在真实机器上调工具、写脚本、跑处理流程
- 结果、审批和下一步又能回到用户最方便的设备上

这让它天然适合那些不常坐在开发环境里、但每天都在被数字琐事消耗的人。

### 5. 输出要优先服务决策与结果，而不是堆细节

用户通常先想知道的是：

- 现在有没有需要我补充或确认的地方？
- 这次到底替我省掉了什么工作？
- 结果能不能直接拿去用？
- 如果还没完成，下一步卡在哪？

所以系统应该优先输出：

- 当前状态
- 已完成结果
- 需要用户介入的事项
- 可选细节，而不是默认把所有执行日志倾倒出来

### 6. `Session`、`App`、并发与分发依然重要，但它们是能力层，不是第一句卖点

这些能力依然成立：

- `Session` 适合承载持久化工作线程
- `App` 适合封装验证过的 workflow
- 并发 session 与跨 session 上下文在某些场景下很有价值
- 分发最终能放大已经验证的能力

但它们不该压过第一阶段更本质的问题：**让更多普通用户第一次真切感受到“AI 帮我把工作自动化了”**。

## 当前产品下注

目前最值得继续加注的方向是：

- AI 主动引导的 intake / problem discovery
- 默认 `Welcome App` + 首次引导，把用户从“我该怎么开始”带到第一个可执行 automation brief
- 围绕简单重复数字工作的高回报自动化用例
- state-first、decision-first 的交互与输出
- 手机发起 + 真机执行 + 跨端回看的顺滑闭环
- 把已验证的 automation 沉淀为可复用 `App` / template

---

## 暂不作为 headline 的方向

- 一次用户请求扇出很多并发 session
- 更重的项目管理 / board / hierarchy 抽象
- 先做“超级个体工作台”的品牌叙事
- 先做大而全的 no-code builder
- 先把 workflow 分发当成第一增长故事

这些方向不是错，而是应该建立在前面那层真实价值被反复验证之后。

---

## 仍然开放的问题

这些问题值得继续观察，但不必现在提前定死：

- 哪些“简单但高回报”的自动化场景最适合做成第一批示例与 onboarding？
- `Welcome App` 第一次应该问哪些信息，既足够帮助 AI 判断问题，又不会把用户带进冗长表单或审问感？
- AI 应该怎样收集样例、目标和约束，既足够聪明，又不会让非专业用户觉得被审问？
- 哪些确认 / 审批节点最适合移动端？
- `App` 应该在什么时机从“内部复用模板”变成“可分享入口”？
- 并发 session 什么时候开始从后台能力变成用户真正会感知并主动需要的东西？
